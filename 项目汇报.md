# 项目汇报

## 项目题目

**AI for BN: 一个面向 BN 相关二维材料的组成级 band gap 预测与候选筛选 PoC**

---

## 1. 摘要

本项目的目标不是直接给出可发表的材料发现结论，而是构建一个能够稳定运行、可复现实验流程、便于后续扩展的 **AI-assisted materials screening proof-of-concept (PoC)**。当前系统以 BN 相关二维材料为切入点，基于公开材料数据库中的 band gap 标签，完成了从数据获取、规范化、主题子集提取、组成级特征构造、基线回归建模、测试集评估、候选材料打分，到结果落盘与网页展示的完整链路。

当前实现使用 JARVIS/2DMatPedia 的 `twod_matpd` 数据集，采用基于化学式的基础组成特征和 `HistGradientBoostingRegressor` 作为默认基线模型。在当前随机行级切分设置下，测试集结果为：

- **MAE = 0.9752**
- **RMSE = 1.3371**
- **R² = 0.1589**

这组结果表明，当前系统已经能够形成一个非空的、可运行的信号通路，但预测能力仍然较弱，不足以支撑高可信度的材料筛选结论。项目现阶段的核心价值，在于完成了一条低复杂度但端到端闭环的 AI 筛选工作流，为后续做更严谨的表示学习、评估协议和候选生成提供了集成基础。

---

## 2. 研究动机与问题定义

### 2.1 从 AI 角度看，这个项目在解决什么问题

从 AI 视角看，本项目关注的是一个典型的 **small-to-medium scale scientific regression + ranking** 问题：

1. 给定一批带有目标性质标签的材料样本；
2. 使用一个可解释、低工程成本的模型学习“材料表示 -> 目标性质”的映射；
3. 将该模型用于候选集合排序，作为后续更高成本验证的前置筛选器。

当前目标性质为 `band_gap`。从机器学习的角度，它是一个明确的实值回归目标，适合用作概念验证中的第一任务。

### 2.2 为什么需要 PoC，而不是一开始追求最强模型

对当前阶段而言，最大的风险不是“模型还不够复杂”，而是“整条系统链路是否成立”。

如果没有一个先跑通的 PoC，就很难回答这些更基础的问题：

- 数据源是否稳定可取？
- 材料记录是否能规范化到统一表结构？
- 候选筛选是否能落到可检查的 artifact？
- 前端展示、notebook 演示和命令行运行能否共存？
- 后续更强特征或更强模型是否有明确插入位置？

因此，本项目优先追求的是 **workflow validity**，而不是当前数值最优。

---

## 3. 为什么选择 BN 相关材料作为 PoC 范围

当前 PoC 不是试图覆盖“所有材料”，而是主动压缩问题范围，选择 BN 相关材料作为起点。这一设计主要基于三点考虑。

### 3.1 主题边界清晰，便于构造垂直切片

相较于泛化到整个材料空间，BN 相关材料提供了一个边界更清晰的主题切口。这样做有利于：

- 在报告中保持问题定义聚焦；
- 在数据层面构造一个可解释的主题子集；
- 在候选生成层面设计一个足够简单但可运行的规则系统。

### 3.2 任务既可讲清楚，又便于扩展

BN 及其相近 III-V 风格体系足够具体，适合作为概念验证对象；同时又保留了向更广材料家族扩展的空间。因此它既适合作为 PoC，也不会把系统设计锁死在单一实例上。

### 3.3 对当前 AI 范式是合理难度

当前实现使用的是组成级特征，而不是结构级表示。如果一开始就选择依赖高复杂度结构表示的更难任务，PoC 的系统风险会显著上升。BN 相关 band gap 预测作为第一版任务，在“可实现性”和“领域相关性”之间形成了比较合适的平衡。

## 4. 当前系统设计

当前系统可以理解为一个轻量分层结构：

- **数据层**：拉取、规范化、缓存、manifest
- **特征与建模层**：公式解析、组成级特征、切分、训练、评估
- **候选筛选层**：候选生成、模型打分、Top-K 输出
- **展示层**：命令行入口、notebook、Streamlit

### 4.1 数据层

数据来自：

- `twod_matpd`
- 获取方式：`jarvis.db.figshare.data()`

标准化后的核心字段为：

- `record_id`
- `source`
- `formula`
- `target`

当前处理后数据规模：

- 总记录数：**6351**
- 唯一 formula 数：**4381**
- 非空目标值：**6351**

系统将数据以两种形式保存：

- `data/raw/twod_matpd.json`：原始缓存
- `data/processed/twod_matpd.parquet`：规范化后的表

同时通过 `manifest.json` 保存数据来源与拉取记录。

### 4.2 主题子集构造

项目通过对 `formula` 解析元素符号，并保留同时包含 `B` 与 `N` 的记录，构造所谓“BN slice”。

当前得到：

- BN 相关切片条数：**12**

需要强调的是，这里的 BN slice 是一个**主题过滤结果**，而不是训练专用数据集。当前模型训练仍然使用整个标准化数据表，而 BN slice 主要服务于主题聚焦和展示。

### 4.3 候选生成层

当前候选生成采用简单枚举策略：

- Group 13: `B, Al, Ga, In, Tl`
- Group 15: `N, P, As, Sb, Bi`
- 两两组合得到 25 个候选 formula

这是一个明显的 PoC 设计：

- 优点是简单、可解释、便于快速验证完整筛选流程；
- 缺点是没有引入稳定性、结构合理性或合成可行性约束。

系统最终保留 `top_k = 20`，因此产物中输出 20 个候选结果。

### 4.4 特征层

当前模型并未使用结构信息，而是采用 **formula-derived composition features**。具体包括：

- `n_elements`
- `sum_z`
- `max_z`
- `min_z`
- `mean_z`
- `contains_B`
- `contains_N`

这组特征本质上编码的是：

- 元素个数
- 原子序数分布的粗粒度统计
- 是否出现关键元素 B/N

一个需要明确说明的事实是：

- 配置中出现了 `use_matminer = True`
- 但当前代码路径中**并没有真正接入 matminer 特征**

因此，当前系统应被准确描述为：

**基于化学式的基础组成特征基线，而非 matminer-enhanced baseline。**

### 4.5 模型层

默认模型为：

- `HistGradientBoostingRegressor`

当前默认参数：

- `max_depth = 6`
- `learning_rate = 0.05`
- `max_iter = 500`
- `min_samples_leaf = 20`
- `random_state = 42`

代码中同时保留了 `RandomForestRegressor` 配置，但默认未启用。

这一选择符合当前 PoC 的工程目标：

- 训练快
- 依赖轻
- 易复现
- 容易在小规模 tabular baseline 上稳定运行

### 4.6 评估层

当前评估方式为随机行级划分：

- train: 0.8
- val: 0.1
- test: 0.1

在当前数据规模下，对应约为：

- train: **5080**
- val: **635**
- test: **636**

评估指标为：

- MAE
- RMSE
- R²

需要指出的是，当前实现虽然构造了 `val` 掩码，但默认训练与输出里更核心的使用场景仍是 train/test，验证集并未被充分用于系统化调参。

## 5. 当前实现的数据流与模型流

### 5.1 数据流

当前 `main.py` 里的主流程是线性的，方便 notebook 迁移与快速审阅：

1. 清理缓存
2. 读取配置
3. 创建运行目录
4. 加载或构建数据集
5. 过滤 BN 子集
6. 生成候选集合
7. 将完整数据表转为特征表
8. 划分 train/val/test
9. 训练基线模型
10. 在 test 上评估
11. 对候选做打分与排序
12. 保存 metrics / predictions / BN slice / screened candidates / parity plot

### 5.2 模型流

模型流可以概括为：

- **输入表示**：formula-derived composition features
- **监督目标**：`band_gap`
- **学习器**：gradient boosting regressor
- **输出一**：test-set regression metrics
- **输出二**：candidate ranking

从 AI 系统角度，这条模型流已经具备了一个最小可工作闭环所需的关键部分：

- 可训练
- 可评估
- 可部署为离线打分器
- 可输出明确 artifact

## 6. 当前结果

### 6.1 定量结果

当前测试集指标：

- **MAE = 0.9752490635938119**
- **RMSE = 1.3370596135896187**
- **R² = 0.15892105776952326**

### 6.2 筛选结果

当前 `screened_candidates.csv` 中位于前列的候选包括：

- `BN` -> 2.515
- `AlP` -> 1.392
- `AlN` -> 1.112
- `GaN` -> 1.068
- `InN` -> 0.799

这些结果说明系统已经可以把“候选生成”与“模型打分”连起来，但目前排序结果仍应被视为 **baseline-generated leads**，而不是高可信候选结论。

### 6.3 这些结果意味着什么

从积极的角度看，当前结果说明：

- 系统并非完全失效，`R²` 为正值，表明存在一定可学信号；
- 工程链路已经闭环，能够持续地产出标准化 artifact；
- 候选筛选不再停留在概念层，而是可以形成实际排序文件。

但从严格的科研角度看，当前结果**绝不应被过度解释**。它们更多意味着：

- 这个问题值得继续做；
- 当前设计是一个有效的起跑线；
- 但还远未到可以支持强科学结论的阶段。

### 7.1 完整性高于复杂度

当前系统最大的优点不是模型复杂，而是 **vertical slice 完整**。它已经具备：

- 数据获取
- 数据清洗与缓存
- 特征构造
- 模型训练与评估
- 候选排序
- 结果展示

这使得后续任何方法升级都有明确挂载点。

### 7.2 工程形态清晰

项目保留了非常清晰的入口分工：

- `main.py`：完整离线流水线
- `main.ipynb`：分步演示与解释

这种形态非常适合教学展示、快速复现实验以及后续模块替换。

### 7.3 基线足够轻，便于快速迭代

由于当前特征和模型都比较轻量，实验成本较低，适合快速验证后续方向，例如：

- 更合理的 split 方案
- 更强的特征表示
- 更严格的候选约束

### 7.4 结果产物标准化程度已经足够支撑研究迭代

当前系统已经保存：

- `metrics.json`
- `predictions.csv`
- `bn_slice.csv`
- `screened_candidates.csv`
- `manifest.json`

这意味着后续可以在不重写基础设施的前提下，直接开展 error analysis、candidate analysis 和 baseline comparison。

## 8. 技术弱点与有效性威胁

这是当前项目最需要诚实面对的部分。

### 8.1 表示能力明显不足

当前特征只依赖 formula，而不包含结构信息。这意味着模型无法区分：

- 同组成但不同结构的材料；
- 更细粒度的局域环境差异；
- 与 band gap 强相关的结构性信息。

因此，当前模型应被归类为 **composition-only baseline**，而非具备强领域表达能力的材料模型。

### 8.2 当前评估协议可能偏乐观

一个更严肃的问题是：

- 当前切分是随机行级切分；
- 数据中存在大量重复 formula；
- 6351 条记录只对应 4381 个唯一 formula；
- 有 3002 条记录属于重复 formula 行。

这意味着训练集和测试集之间可能共享相同 formula，从而引入信息泄漏风险。对于当前这种完全基于 formula 的特征表示，这一点尤其重要，因为模型几乎直接建立在 formula 上。

换句话说，当前 test metrics 很可能**高估了真实泛化能力**。

### 8.3 候选生成缺乏物理或化学约束

当前候选生成只是简单枚举 Group 13 × Group 15，并不保证：

- 热力学稳定性
- 合理结构存在性
- 实验可合成性
- 与真实二维材料分布的一致性

因此，当前排序结果只能作为“模型在简单候选空间上的打分”，而不能作为可靠的 discovery claim。

### 8.4 未建模不确定性

当前系统只输出点预测，没有：

- predictive uncertainty
- ensemble variance
- calibration analysis

这会削弱候选排序的决策价值，因为实际 screening 往往更需要“高值且高可信”的候选，而不是仅有单点分数。

### 8.5 训练与推理尚未完全解耦

当前系统仍是训练和离线筛选紧耦合。尚未形成独立的：

- 模型持久化 artifact
- 特征 schema 版本控制
- 统一推理接口

这会限制后续将 notebook、Streamlit、API 共享到同一 inference backend 的能力。

### 8.6 部分配置项尚未真正落地

当前代码里保留了 `matminer`、`api`、`llm` 等配置位，但这些并不是已经上线的功能。若在汇报中不加区分，容易给人“系统能力被高估”的印象。

## 9. 对当前成果的谨慎判断

如果要用一句学术上比较诚实的话概括当前状态，我会这样表述：

> 当前工作完成了一个以 BN 相关二维材料为主题的、组成级 band gap 预测与候选筛选 PoC。其主要贡献不在于提出了一个高性能模型，而在于打通了从公开材料数据到候选排序 artifact 的完整 AI 工作流，并暴露出了后续研究中最关键的方法学短板。

这意味着本项目当前的价值主要体现在三方面：

1. **系统价值**：完整链路已经建立，后续方法升级有明确落点；
2. **方法价值**：composition-only baseline 为更强模型提供对照基线；
3. **研究价值**：项目已经清楚暴露出“评估协议”和“表示能力”是下一阶段的主要瓶颈。

但它当前并不构成：

- 强预测器
- 强候选发现系统
- 可直接支持实验决策的生产级工具

## 10. 下一阶段计划与里程碑

为了把当前 PoC 推进到更有研究价值的原型，我认为最合理的路线不是直接堆更复杂模型，而是按以下顺序演进。

### Milestone 1: 修正评估协议，建立可信基线

目标：先让指标变得可信。

建议工作：

- 将随机行级 split 改为按 `formula` 分组的 split；
- 进一步考虑 composition-grouped 或 scaffold-like split；
- 加入 cross-validation；
- 输出更加系统化的 error analysis。

预期结果：

- 得到一个更可信但可能更低的基线分数；
- 明确当前 baseline 到底有多少真实泛化能力。

### Milestone 2: 升级表示层

目标：从弱组成特征过渡到更合理的材料表示。

建议工作：

- 真正接入 `matminer` 的 composition features；
- 如果数据条件允许，逐步引入结构特征；
- 形成 composition-only 与 richer-feature baseline 的对比实验。

预期结果：

- 建立第一轮真正有研究意义的 baseline comparison；
- 判断性能瓶颈主要来自表示层还是模型层。

### Milestone 3: 解耦训练与推理，增强系统可用性

目标：让系统从“脚本化流程”变成“可复用原型”。

建议工作：

- 持久化训练模型和特征 schema；
- 提供统一 `predict_formulas()` 推理入口；
- 让 notebook、Streamlit 和未来 API 共用同一推理后端。

预期结果：

- 系统结构更清晰；
- 展示和实验分析不再依赖重复训练。

### Milestone 4: 提升候选筛选可信度

目标：让 screening 更接近“研究工具”而不只是 demo。

建议工作：

- 对候选生成加入规则约束；
- 增加 uncertainty-aware ranking；
- 对 Top candidates 加入二次过滤逻辑。

预期结果：

- 让候选排序更接近可讨论的研究输出；
- 降低“高分但无意义候选”的比例。

## 11. 坦诚评估

如果导师问我：“这个项目现在最值得认可的地方是什么？最不能回避的问题又是什么？”

我的回答会是：

### 最值得认可的地方

- 它不是空想，而是已经有真实运行结果的 end-to-end prototype；
- 它把数据、模型、筛选、展示串成了一条可复现链路；
- 它在工程上为后续严肃研究做了足够好的铺垫。

### 最不能回避的问题

- 当前模型仍然太弱；
- 当前评估协议不够严谨，存在明显泄漏风险；
- 当前候选生成过于简化，不能承载强结论；
- 当前结果更多证明“系统可运行”，而不是“科学命题已被验证”。

所以我不会把当前工作包装成“已经取得了重要材料发现”，而会把它定位为：

**一个已经完成系统验证、但尚未完成方法验证的 AI-for-materials PoC。**

这个定位虽然不夸张，但更稳，也更容易获得严谨导师的认可。

## 12. 结论

本项目当前最清晰的贡献是：

- 以 BN 相关二维材料为主题，完成了一个组成级 band gap 预测与候选筛选的可运行原型；
- 实现了从公开数据源到候选排序 artifact 的端到端链路；
- 提供了一个后续可升级的研究骨架。

当前结果尚不支持强科学结论，但已经足以支撑下一阶段的严谨研究工作。若后续按照“先修评估、再升表示、再做推理解耦与候选约束”的路线推进，这个项目有潜力从演示型 PoC 逐步进化为一个更有方法学价值的研究原型。
