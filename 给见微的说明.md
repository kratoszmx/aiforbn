# 给见微的说明

## 先用一句话说清楚

这个项目本质上是在做一件事：

**先从公开二维材料数据里拿到一批样本，再用一个很轻量的机器学习基线模型去预测材料的 band gap，最后把一组 BN 风格候选材料按预测结果排序，形成一个可演示的 AI 初筛流程。**

如果你只想先抓住核心，可以把它理解成：

- 不是在“发现新材料”
- 而是在做一个 **AI 辅助材料初筛的 PoC**
- 重点是把流程跑通，而不是现在就把精度做到很强

---

## 1. 这个项目到底要解决什么问题

这个项目想回答的问题可以写得很朴素：

> 如果我已经有一批公开材料数据，能不能先训练一个简单模型，让它学一点材料组成和 band gap 之间的关系，然后用这个模型去快速给一批 BN 相关候选材料做初步打分？

这件事的价值在于，它把一个原本比较散的想法落成了一条完整流程：

1. 拿数据
2. 整理数据
3. 筛出和 BN 相关的部分
4. 把材料化学式变成模型可读特征
5. 训练模型
6. 评估模型
7. 对候选材料排序
8. 把结果保存并展示出来

所以它现在更像是一个“**方法链路验证项目**”，而不是一个“已经完成科研结论的项目”。

---

## 2. 为什么选 BN，为什么选 band gap

### 2.1 为什么是 BN

这里的 BN 指的是 **boron nitride，硼氮化物**。当前项目把范围稍微放宽了一点，不只看严格的 BN 本体，也看“和 BN 思路相近的一组 III-V 风格材料候选”。

选 BN 这个主题，主要因为它有几个优点：

- **主题比较聚焦**，不是对整个材料宇宙胡乱建模
- **对外比较好讲**，即使听众不懂化学，也能理解“围绕某类材料做 AI 初筛”
- **容易扩展**，后面可以从 BN 扩到更多 III-V 材料、更多特征、更多筛选规则

换句话说，BN 是一个很适合做第一版 demo 的切入口。

### 2.2 为什么选 band gap

当前预测目标是 `band_gap`。

你可以先把它理解为一个重要材料性质，很多时候它跟材料更偏绝缘、半导体、还是更容易导电有关。就算先不深究物理细节，也可以知道：

- 它是一个常见、可量化的目标
- 在公开材料数据库里比较常见
- 很适合做监督学习里的回归任务

所以从 AI 角度看，`band_gap` 是一个合理的第一目标：

- 有数据
- 有明确标签
- 容易定义任务
- 容易做基线模型

## 3. 数据从哪里来，数据长什么样

### 3.1 数据来源

当前数据源是：

- **2DMatPedia / twod_matpd**
- 通过 **JARVIS-Tools** 运行时拉取

项目第一次跑时会下载数据，并做两层落盘：

- `data/raw/twod_matpd.json`：原始缓存
- `data/processed/twod_matpd.parquet`：处理后的标准表
- `data/processed/manifest.json`：记录数据集来源和拉取信息

### 3.2 当前标准化后的数据长什么样

在代码里，原始数据最后会被整理成一个很简单的表，核心列只有：

- `record_id`
- `source`
- `formula`
- `target`

也就是说，这个 PoC 最终真正喂给后续流程的数据，是一种很简化的材料表：

- 每一行是一条材料记录
- `formula` 是化学式
- `target` 是目标值，这里就是 band gap

### 3.3 当前数据规模

基于当前已处理数据：

- 总样本数：**6351**
- 唯一化学式数：**4381**
- `target` 非空：**6351**
- BN 相关切片：**12** 条

这里有一个你之后一定要记住的细节：

- 6351 条记录里，只有 4381 个唯一 formula
- 说明存在不少“相同 formula 对应多条记录”的情况

这会直接影响评估可信度，后面我会专门讲。

## 4. 整条 pipeline 到底在做什么

这部分最重要。你只要把 `main.py` 读明白，基本就理解了整个项目。

`main.py` 的主流程是：

1. 清缓存
2. 读配置
3. 初始化目录
4. 加载或构建数据集
5. 过滤 BN 子集
6. 生成候选材料
7. 构造特征表
8. 划分 train/val/test
9. 训练基线模型
10. 在 test 上评估
11. 对候选材料做预测排序
12. 保存结果文件和图

下面逐步拆开。

### 4.1 读取配置

配置文件在：`configs/default.py`

里面定义了这几个关键信息：

- 数据集：`twod_matpd`
- 目标列：`band_gap`
- 划分比例：`0.8 / 0.1 / 0.1`
- 模型类型：`hist_gradient_boosting`
- 候选筛选保留数量：`top_k = 20`

所以这不是一个“全自动自己决定实验设置”的项目，而是一个**配置驱动的 PoC**。

### 4.2 加载或构建数据集

对应代码：`src/pipeline/data.py`

它会做的事情是：

- 如果 `data/processed/twod_matpd.parquet` 已经存在，就直接读取
- 否则从 JARVIS 拉原始数据
- 从原始条目里提取 formula 和 target
- 规范成统一表结构
- 存 parquet 和 manifest

输入：
- 原始 JARVIS 数据

输出：
- `dataset_df`
- `manifest`

这里还有一个实际修复过的 bug：

- 原来代码会过早把 `jid` 当成 formula
- 现在已经改成优先从真正的 formula / atoms / composition 里拿材料式，再退回 `jid`

这个修复是对的，因为 `jid` 本质是记录标识，不应该优先当材料化学式。

### 4.3 过滤 BN 子集

对应代码：`filter_bn()`，在 `src/pipeline/features.py`

它的逻辑非常直接：

- 从 `formula` 提取元素符号
- 判断一个记录是否同时包含 `B` 和 `N`
- 如果包含，就保留

输入：
- `dataset_df`

输出：
- `bn_df`

当前跑出来的 BN 切片有 **12 条**。

注意，这里的“BN 子集”不是只保留纯 BN，而是所有 **formula 中同时包含 B 和 N 的样本**，所以像 `BC2N`、`Si2BN` 这种也会进来。

### 4.4 生成候选材料

对应代码：`generate_bn_candidates()`

现在的候选生成规则是一个**非常明显的演示型规则**：

- group 13: `B, Al, Ga, In, Tl`
- group 15: `N, P, As, Sb, Bi`
- 两两组合，得到 25 个候选 formula

所以候选空间本质上是：

**5 × 5 = 25 个简单的 III-V 风格组合**

例如：

- BN
- AlN
- GaN
- AlP
- TlBi

然后筛选阶段只保留 top 20，因此最终 artifact 里是 **20 条候选结果**。

这一步你一定要理解清楚：

- 它不是基于化学稳定性、结构生成或第一性原理去提出候选
- 它只是一个**很轻量的枚举式候选生成器**
- 目的主要是为了让整条 AI screening pipeline 有东西可筛

### 4.5 构造特征表

对应代码：`build_feature_table()` + `_basic_features()`

当前特征完全是**基于 formula 的基础组成特征**，没有用晶体结构。

实际用到的特征包括：

- `n_elements`：元素个数
- `sum_z`：原子序数总和
- `max_z`：最大原子序数
- `min_z`：最小原子序数
- `mean_z`：平均原子序数
- `contains_B`：是否含 B
- `contains_N`：是否含 N

也就是说，模型现在看到的其实只是：

> 这个材料由哪些元素构成，这些元素的原子序数大概落在什么区间

这是一个非常基础的表示方式。

要特别注意一个“看起来像有、其实还没有”的点：

- 配置里有 `use_matminer = True`
- 但当前代码**并没有真正调用 matminer 特征**
- 真正落地的仍然只有上面这些 basic composition features

所以如果以后有人问“这个项目用了 matminer 吗”，当前准确答案是：

**配置上留了口子，但实际实现还没有接进去。**

### 4.6 划分训练集、验证集、测试集

对应代码：`make_split_masks()`

当前做法是：

- 随机打乱行索引
- 按比例切成 train / val / test
- 比例是 0.8 / 0.1 / 0.1

当前样本规模下，对应大致是：

- train：**5080**
- val：**635**
- test：**636**

但这里有一个很重要的问题：

- 当前切分是**按行随机切**
- 不是按 formula 分组切
- 数据里又存在大量重复 formula

这意味着：

- 同一个 formula 可能同时出现在 train 和 test
- 评估结果可能偏乐观

这属于当前项目最真实、最严肃的局限之一。

### 4.7 训练模型

对应代码：`train_baseline_model()` + `make_model()`

当前默认模型是：

- `HistGradientBoostingRegressor`

参数来自配置：

- `max_depth = 6`
- `learning_rate = 0.05`
- `max_iter = 500`
- `min_samples_leaf = 20`
- `random_state = 42`

代码里也预留了 `RandomForestRegressor` 的配置，但默认没有用它。

所以当前可以准确地说：

**这是一个基于组成特征的 sklearn 梯度提升回归基线。**

### 4.8 评估模型

对应代码：`evaluate_predictions()`

评估是在 test 集上做，输出三个指标：

- `mae`
- `rmse`
- `r2`

当前 artifact 中的结果是：

- **MAE = 0.9752490635938119**
- **RMSE = 1.3370596135896187**
- **R² = 0.15892105776952326**

怎么理解这组数：

- `R² > 0`，说明模型不是完全没学到信号
- 但 `R² ≈ 0.159`，说明它离“强模型”还很远
- 这更像是一个**能跑通的弱基线**，而不是一个结果已经很漂亮的科研模型

### 4.9 对候选材料做筛选

对应代码：`screen_candidates()`

这一步会：

- 把生成的候选 formula 也转成同样的 basic features
- 用训练好的模型去预测它们的 target
- 按预测值排序
- 保留 top 20

当前 `screened_candidates.csv` 里排前面的有：

- `BN`：2.515
- `AlP`：1.392
- `AlN`：1.112
- `GaN`：1.068
- `InN`：0.799

但这组结果的正确理解是：

- 它们只是**在当前弱基线 + 当前简化特征 + 当前简单候选生成规则下的排序结果**
- 不是“发现了新的最好材料”
- 也不是“可以直接指导实验”的可靠结论

## 5. 当前模型实际上学了什么，没有学什么

### 当前模型学到的东西

它主要学的是：

- 某些元素组合的大致统计特征
- 这些统计特征和 target 之间的粗略相关性

换句话说，它更像在学：

> “看到某类元素组合时，band gap 大概会落在哪个区间”

### 当前模型没有学到的东西

它没有真正学到：

- 晶体结构
- 原子排布
- 键长、配位环境
- 电子结构
- 稳定性
- 可合成性

所以它不是一个真正“懂材料结构”的模型，只是一个**组成级 baseline**。

### `artifacts/metrics.json`

存评估指标，也就是这次训练在 test 集上的 MAE / RMSE / R²。

存 BN 相关子集，也就是包含 B 和 N 的材料记录。

- 证明项目确实围绕 BN 做了聚焦切片

### `artifacts/screened_candidates.csv`

存候选筛选结果，也就是对那批 BN 风格候选材料的打分与排序。

是真值和预测值的对比图。

它有展示价值，但当前你如果只是要和 GPT Pro 模拟导师对话，其实**文本结果更重要**。

说明这次数据使用的来源和拉取记录。

## 9. 当前最真实的局限是什么

这部分你一定要记住，因为以后老师或 GPT Pro 很可能会问。

### 9.1 特征太简单

现在的特征只看 formula，不看结构。

这意味着模型表达能力天然很弱。

### 9.2 切分方式不够严谨

当前是按行随机切，不是按 formula 分组切。

而数据里重复 formula 又不少，这会带来**信息泄漏风险**，让 test 分数可能显得比真实泛化能力更好。

### 9.3 候选生成太演示化

当前候选只是 5×5 枚举组合，不考虑更深层的化学合理性。

### 9.4 没有不确定性估计

现在输出只有 prediction，没有 confidence / uncertainty。

这会让候选排序显得过于“像结论”，但其实它只是一个粗排序。

### 9.5 没有单独保存模型 artifact

当前保存了结果表和图，但没有把训练后的 model 连同 feature schema 一起持久化成一个标准 inference artifact。

### 9.6 Streamlit 只是展示，不是系统化应用

它还没有：

- 实时预测输入框
- 可调筛选条件
- 多页面诊断
- 独立 inference API

### 9.7 配置里有些“未来功能”还没真正实现

例如：

- `use_matminer = True`
- `api.enabled`
- `llm.enabled`

这些现在更多是预留位，而不是已经做完的功能。

## 10. 如果继续做，最值得优先补什么

如果你问我“下一步最值的 3 到 5 件事是什么”，我的排序是：

### 第一优先级：先把评估做严谨

最值得优先补的是：

- 按 formula 分组切分
- 或者按 composition / scaffold 做更合理的 split
- 最好加交叉验证

这是因为如果评估不严谨，后面再加花哨模型意义也有限。

### 第二优先级：补更强的特征

例如：

- 真正接入 matminer
- 加结构相关特征
- 至少做 composition-only baseline vs richer-feature baseline 对比

### 第三优先级：保存模型和独立推理接口

把“训练”和“推理”分开：

- 保存 model artifact
- 保存 feature schema
- 做一个 `predict_formulas()` 之类的清晰入口

这样 notebook、Streamlit 和未来 API 就能共用同一套推理逻辑。

### 第四优先级：让候选筛选更可信

例如：

- 不只是简单枚举
- 增加更合理的材料约束
- 对高分候选做二次筛选

### 第五优先级：把 Streamlit 从 viewer 升级成 demo app

可以加：

- 输入 formula 即时预测
- 候选筛选页
- 指标诊断页
- BN 子集专题页
